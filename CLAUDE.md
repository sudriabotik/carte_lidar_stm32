## Project Overview

This is an STM32G431 embedded firmware project for a LIDAR-based robot control card. The system reads data from an LD06 LIDAR sensor via UART/DMA, processes the point cloud data, and communicates with other robot nodes via CANopen protocol.

**Target MCU**: STM32G431RBTx (ARM Cortex-M4)
**Key Features**:
- LD06 LIDAR data acquisition using DMA-based double buffering
- Point cloud processing and obstacle detection
- CANopen communication stack (CANopenNode)
- Integration with custom robotics library (lib_robot submodule)

## Build System

This project supports both **CMake** and **STM32CubeIDE** build systems.

### CMake Build (Recommended)

```bash
# Configure for Debug build
cmake --preset Debug

# Build
cmake --build build/Debug

# Configure for Release build
cmake --preset Release
cmake --build build/Release
```

The build output will be in `build/Debug/carte_lidar.elf` or `build/Release/carte_lidar.elf`.

### STM32CubeIDE Build

Open the `STM32CubeIDE` folder in STM32CubeIDE and build through the IDE.

## Flashing and Debugging

### Flash the firmware
```bash
./flash_debug.sh
```
This uses OpenOCD with ST-Link to program `build/Debug/carte_lidar.elf`.

### Start GDB debug server
```bash
./start_debug.sh
```
This starts OpenOCD for use with GDB or IDE debuggers.

## Project Structure

### Source Code Organization

- **Core/**: STM32 HAL initialization and peripheral configuration
  - `Core/Src/main.c`: Main application entry point and event loop
  - `Core/Src/stm32g4xx_it.c`: Interrupt handlers
  - `Core/Src/dma.c`, `fdcan.c`, `usart.c`, `tim.c`, `gpio.c`: Peripheral initialization (auto-generated by STM32CubeMX)

- **User/**: Application-specific LIDAR processing code
  - `User/Src/lidar_reader.c`: UART/DMA double-buffer management, frame parsing
  - `User/Src/lidar_data_processor.c`: Point cloud filtering and blob detection
  - `User/Src/lipkg_copy.c`: LD06 protocol frame parsing
  - `User/Src/canopen_cmd.c`: CANopen application interface

- **CANopenNode/**: CANopen stack (git submodule)
- **CANopenNode_STM32/**: STM32 hardware driver for CANopenNode
  - `OD.h`, `OD.c`: CANopen Object Dictionary (auto-generated, do not edit manually)
  - `CO_app_STM32.c`: Application-level CANopen initialization
  - `CO_driver_STM32.c`: Low-level FDCAN driver

- **lib_robot/**: Shared robotics library (git submodule from sudriabotik)
  - Provides coordinate math (`coords.h`), motor control, PID controllers, etc.

- **Drivers/**: STM32 CMSIS and HAL libraries (vendor code)

### Configuration Files

- **carte_lidar.ioc**: STM32CubeMX project file (peripheral configuration)
- **CMakeLists.txt**: Main CMake build configuration
- **CMakePresets.json**: CMake preset definitions (Debug/Release)
- **STM32G431XX_FLASH.ld**: Linker script
- **cmake/gcc-arm-none-eabi.cmake**: ARM GCC toolchain configuration

## Architecture Details

### LIDAR Data Flow

1. **DMA Reception**: UART2 receives LD06 data into dual buffers (`lidar_rx_buffer_1`, `lidar_rx_buffer_2`)
2. **Buffer Swapping**: `HAL_UART_RxCpltCallback` swaps buffers when DMA completes, sets `lidar_rx_buffer_new` flag
3. **Frame Parsing**: `lidar_process_buffer()` searches for LD06 frame headers and validates CRC
4. **Point Accumulation**: Valid measurements are converted to Cartesian coordinates and stored in `lidar_points_buffer[]`
5. **360° Scan Detection**: Detects when LIDAR completes a full revolution by tracking angle wraparound
6. **Processing**: Once a full scan is complete, `lidar_processor_find_blob()` identifies obstacles

### LD06 LIDAR Protocol

- **Frame size**: 47 bytes
- **Header byte**: 0x54
- **Data format**: Little-endian
- **Frame structure**: header (1) + length (1) + speed (2) + start_angle (2) + measurements (3 × 12) + stop_angle (2) + timestamp (2) + crc (1)
- **Each measurement**: distance (2 bytes) + confidence (1 byte)
- **Angles**: Stored in hundredths of degrees (e.g., 9000 = 90.00°)

See detailed documentation in [User/Inc/lidar_reader.h](User/Inc/lidar_reader.h).

### CANopen Integration

The system uses CANopenNode stack with FDCAN1 peripheral:
- **Node ID**: 3 (configured in `main.c`)
- **Baudrate**: 500 kbps
- **Object Dictionary**: Defined in `CANopenNode_STM32/OD.h` (auto-generated from `DS301_profile.eds`)
- **Current status**: CANopen processing is commented out in main loop (`canopen_app_process()`) due to DMA interference issues

**Known Issue**: Recent commits indicate CANopen stack interferes with DMA reception. The basic DMA/LIDAR scan works when CANopen is disabled. This is an active debugging area.

### Coordinate System

The `lib_robot` submodule provides coordinate transformation utilities:
- `coords.h`: 2D point math, angle normalization, coordinate frame transformations
- Used to convert LIDAR polar coordinates (distance, angle) to Cartesian (x, y)
- World coordinates are computed relative to robot pose for obstacle avoidance

## Key Files to Modify

When adding features or fixing bugs:

1. **LIDAR processing logic**: Edit `User/Src/lidar_reader.c` and `User/Src/lidar_data_processor.c`
2. **CANopen behavior**: Edit `User/Src/canopen_cmd.c` and potentially `CANopenNode_STM32/CO_app_STM32.c`
3. **Peripheral configuration**: Use STM32CubeMX to edit `carte_lidar.ioc`, then regenerate code
4. **Build configuration**: Edit `CMakeLists.txt` to add new source files or include paths
5. **Object Dictionary**: Use CANopenEditor to modify the EDS file, then regenerate `OD.h`/`OD.c`

## Important Notes

- **DO NOT** manually edit files in `Core/Src/` or `Core/Inc/` outside of `USER CODE` blocks - these are regenerated by STM32CubeMX
- **DO NOT** manually edit `CANopenNode_STM32/OD.h` or `OD.c` - regenerate from EDS file
- UART1 is used for `printf()` debugging output (retargeted via `__io_putchar`)
- UART2 is dedicated to LIDAR communication
- Timer TIM6 is used by CANopen stack for timebase
- The project uses git submodules - remember to `git submodule update --init --recursive` after clone
